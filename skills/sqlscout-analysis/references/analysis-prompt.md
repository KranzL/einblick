## Instructions

1. For each pattern, determine:
   - What the query does in business terms (interpret the SQL purpose)
   - Whether it is a materialization, denormalization, clustering, CDC, or other optimization candidate
   - The specific recommendation with ready-to-execute DDL
   - The expected impact (HIGH/MEDIUM/LOW)
   - The implementation risk and what could go wrong

2. Group related patterns that touch the same tables or serve the same business purpose. A single recommendation may address multiple patterns.

3. Identify patterns you should NOT recommend optimizing (too infrequent, too cheap, or insufficient information to make a confident recommendation). Briefly explain why you're skipping them.

4. Produce a final report with these sections:

   **Executive Summary**: 2-3 sentences on the biggest findings and potential savings. Lead with the single highest-impact recommendation. Call out the top offending users or warehouses if they stand out.

   **Biggest Offenders**: Analyze the offender data. For each category:
   - Users by cost: Who is burning the most credits? Is it justified by query volume or are they running inefficient queries?
   - Users by runtime: Who has the longest-running queries? Are they scanning too much data or using undersized warehouses?
   - Warehouses: Are any warehouses oversized for the queries they run? Could queries be consolidated or moved to smaller warehouses?
   - Slowest patterns: What makes these queries slow? Can they be optimized with clustering, materialization, or query rewrites?
   - Most data scanned: Are these full table scans that could benefit from clustering keys or column pruning?

   **Top Recommendations**: Each with:
   - Description of what to build and why
   - Which query patterns it addresses (by fingerprint)
   - DDL appropriate to the platform and pattern: CREATE TABLE AS SELECT for materialization, CREATE VIEW for lightweight denormalization, CREATE DYNAMIC TABLE (Snowflake) or Delta Live Table (Databricks) for incremental refresh, ALTER TABLE CLUSTER BY or OPTIMIZE ZORDER for data layout
   - **Original query** (representative form of the current pattern)
   - **Optimized query rewrite** with inline comments explaining changes. Apply patterns from the "Query Rewrite Patterns" section of the system prompt: predicate pushdown, SELECT * elimination, ROW_NUMBER -> QUALIFY, NOT IN -> NOT EXISTS, APPROX_COUNT_DISTINCT, etc. If no rewrite is worthwhile, explicitly say "Query is already near-optimal" and explain why. Skip this when the query is auto-generated by a BI tool (flag that the fix has to be made in the tool, not the SQL).
   - Estimated impact (executions eliminated, bytes saved, credits saved)
   - Refresh strategy (Task, Dynamic Table, dbt, manual)
   - Priority (HIGH/MEDIUM/LOW)
   - Implementation risks and prerequisites

   **Query Rewrites**: A dedicated section collating the top 5-10 query rewrites extracted from the Top Recommendations above, for teams that want to roll rewrites out independently of materialization. Format each as: fingerprint reference, original SQL, rewritten SQL, one-line summary of the win. Users should be able to copy the rewritten SQL into their dbt models, BI tools, or ad hoc queries immediately.

   **Patterns Skipped**: Patterns you reviewed but chose not to recommend action on, with brief reasoning.

   **Cost Analysis**: Total credits across all analyzed patterns, top credit consumers, and estimated savings if recommendations are implemented. Break down savings by recommendation.

   **Access Patterns**: Summary of which users/roles query which tables. Flag any unexpected access patterns, governance concerns, or cross-schema access that suggests missing permissions or role consolidation.

   **Implementation Checklist**: Numbered priority list with:
   - What to build first, second, third
   - Required permissions (GRANT statements needed)
   - Dependencies between recommendations (e.g., "build the staging table before the dynamic table that references it")
   - What to validate after each implementation (queries to confirm the optimization is working)

5. **Emit structured proposals via the `emit_dbt_proposals` tool** (tool call, not prose). After you write the report text above, call the `emit_dbt_proposals` tool exactly once with machine-readable versions of your Top Recommendations. sqlscout uses these downstream to optionally create dbt model files and open a PR on the user's behalf. The rendered output of your tool call will be appended to the report as a "Proposed dbt Changes" section -- do not duplicate that section in your prose.

   Rules for the tool call:
   - Use `new_model` for brand-new models you're proposing.
   - Use `modify_existing` ONLY if you know the target model exists in the user's dbt project (you'll know from context passed in). Do not guess at names.
   - Use `access_pattern` ONLY if you know a model exists that users should be querying instead. This also requires dbt context. Skip it otherwise.
   - Populate `metrics_addressed` / `patterns_affected` with the fingerprints of the sqlscout patterns each proposal would cover. This is what lets the user's diff tool mark the pattern as "addressed" after implementation.
   - If no concrete proposals fit (small run, nothing worth proposing), call the tool with an empty `proposals: []` array, or skip the tool call entirely.
   - Keep the prose Top Recommendations section as the human-readable narrative. The tool call is the structured data for automation.
